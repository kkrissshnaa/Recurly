Authentication Implementation Report
A Technical Overview of Clerk Authentication Integration & Expo Go Compatibility Fixes

1. Feature Implementation Overview

To implement a secure, premium, and brand-native authentication flow for Recurly, we integrated Clerk Authentication into the Expo Router workspace. The implementation was structured into three primary architectural segments:

Secure Caching Mechanism

Created lib/tokenCache.ts using expo-secure-store. It encrypts and persists Clerk session tokens inside the device's iOS Keychain or Android Keystore, preventing token leaks.

Root Layout Provider Gate

Configured app/_layout.tsx with ClerkProvider and ClerkLoaded. Wrapped the stack router with an InitialLayout guard to check the session state and redirect automatically.

Mockup-Aligned Auth UI

Designed custom, production-grade screens matching the mockup layout, fonts, and warm-cream theme. Handled email/password login, registration, and verification inline.

Dashboard & Settings Hooking

Bound Clerk's useUser() hook to the dashboard index to display the real name/avatar. Hooked settings actions to log out using useClerk().signOut().

2. Step-by-Step Error Resolutions

During compilation and runtime testing inside the Expo Go client, multiple dependency mismatches and syntax issues were identified and successfully resolved. Below is the step-by-step log of each error and its fix:

Error 1: JSON Syntax Error in config
Stray closing brace in app.json

Context: Adding expo-secure-store and @clerk/expo native plugins to the plugins array created syntax conflicts inside the font configuration brackets.

Diagnostic: Line 50 of app.json contained ]}]},. The extra closing brace } invalidated the JSON syntax, crashing the Metro bundler immediately on startup.

Resolution: Modified the brackets on line 50 to ]}] to correctly terminate the font asset list and the plugin sub-array, making the configuration parser-safe.

Error 2: React JSX Compiler Lint warning
Unescaped apostrophe in sign-in redirect text

Context: Adding text items with standard apostrophes in React components caused compile-time warnings.

Diagnostic: The linter flagged <Text>Don't have an account?</Text> in signin.tsx because standard unescaped quotes violate the react/no-unescaped-entities lint rule.

Resolution: Wrapped the text inside braces and double quotes: {"Don't have an account?"} to satisfy the JSX compiler.

Error 3: Expo Go Runtime Crash (Native Module 'ClerkExpo')
Missing native module 'ClerkExpo' in Expo Go

Context: The application compiled but crashed immediately with a red screen on mobile devices running Expo Go.

Diagnostic: The new @clerk/expo SDK (v3+) uses a custom native module called ClerkExpo written in SwiftUI/Jetpack Compose. Since Expo Go is a pre-compiled client, it cannot load custom native modules. This crashes the app at startup, even if only JavaScript hooks are used.

Resolution: Removed @clerk/expo, and installed the pure-JavaScript SDK @clerk/clerk-expo. Since it has no native modules, it works seamlessly inside standard Expo Go. Restructured the hooks in all layouts and screens to align with the core JS-only hooks API (e.g., signIn.create, setActive).

Error 4: Expo Go Runtime Crash (Native Module 'ExpoCryptoAES')
Missing native module 'ExpoCryptoAES' in Expo Go SDK 54

Context: After resolving the Clerk native module crash, the application launched but crashed when initializing authentication.

Diagnostic: Running npm install @clerk/clerk-expo pulled the latest versions of its dependencies: expo-crypto@56.0.4 and expo-auth-session@56.0.14. These require the ExpoCryptoAES native module introduced in Expo SDK 56. Mismatching with the phone's Expo Go SDK 54 client caused it to crash.

Resolution: Ran npx expo install expo-auth-session expo-crypto. This locked both packages to the compatible versions matching Expo SDK 54 (expo-crypto@~15.0.9 and expo-auth-session@~7.0.11), clearing all native runtime crashes.

Error 5: Keyboard Covering Inputs
Password Input Covered on Keyboard Enable

Context: Entering credentials on mobile keyboards (especially on Android and small screens) pushed the password input field off-screen or behind the keyboard, preventing the user from seeing their input.

Diagnostic: The combination of KeyboardAvoidingView with behavior="height" on Android and centering items using flex-1 justify-center inside the child of ScrollView restricted the scroll height. When the keyboard opened, the layout centered the elements within the tiny remaining height, hiding the top elements and overlapping the inputs.

Resolution: Configured KeyboardAvoidingView with behavior={Platform.OS === 'ios' ? 'padding' : undefined} and keyboardVerticalOffset={Platform.OS === 'ios' ? 64 : 0}. Removed flex-1 from the inner View, and applied contentContainerClassName="flex-grow justify-center py-6" directly to the ScrollView. This allows the layout to shrink dynamically and remain naturally scrollable when focused.

3. Core Implementation Files

Below are key structural snippets of our clean, Expo Go compatible implementation:

Root Layout Configuration (app/_layout.tsx)

import { ClerkProvider, ClerkLoaded, useAuth } from "@clerk/clerk-expo";
import { tokenCache } from "@/lib/tokenCache";

export default function RootLayout() {
  return (
    <ClerkProvider publishableKey={publishableKey} tokenCache={tokenCache}>
      <ClerkLoaded>
        <InitialLayout />
      </ClerkLoaded>
    </ClerkProvider>
  );
}

Mockup Sign-In Screen Logic (app/(auth)/signin.tsx)

const { signIn, setActive, isLoaded } = useSignIn();

const handleSignIn = async () => {
  if (!isLoaded) return;
  const result = await signIn.create({
    identifier: email,
    password,
  });
  if (result.status === "complete") {
    await setActive({ session: result.createdSessionId });
  }
};
